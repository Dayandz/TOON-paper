# TOON vs JSON — Token Optimization Experiment for AI Systems

[![License: MIT](https://img.shields.io/badge/License-MIT-green.svg)](https://opensource.org/licenses/MIT)

Repository for the empirical experiment presented in the paper:

> **"TOON as a Replacement for JSON in Artificial Intelligence Projects for Token Usage Optimization"**  

This experiment compares token consumption between JSON and TOON (Token-Optimized Object Notation) across 500 instances distributed over five AI use-case categories, using `llama3.1:8b` running locally via Ollama — no external API or paid service required.

## Installation

### 1. Clone the repository

```bash
git clone https://github.com/Dayandz/TOON-paper.git
cd TOON-paper
```

### 2. Create a virtual environment (recommended)

```bash
# Windows
python -m venv .venv
.venv\Scripts\activate

# Linux / macOS
python3 -m venv .venv
source .venv/bin/activate
```

### 3. Install Python dependencies

```bash
pip install tiktoken nltk rouge-score requests matplotlib
```

> **Note for Windows system Python:** if you get permission errors, add `--break-system-packages` to the command.

Verify the installation:

```bash
python -c "import tiktoken, nltk, rouge_score, requests, matplotlib; print('All dependencies OK')"
```

### 4. Install Ollama

Download the installer for your platform from [https://ollama.com/download](https://ollama.com/download) and run it.

Verify the installation:

```bash
ollama --version
```

### 5. Download the llama3.1:8b model

```bash
ollama pull llama3.1:8b
```

> The download is approximately **4.7 GB** and may take 5–20 minutes depending on your connection speed.

Confirm the model is ready:

```bash
ollama list
# Expected output: llama3.1:8b is listed
```

### 6. Download NLTK resources

Run the following once from a Python shell or script:

```python
import nltk
nltk.download('punkt')
nltk.download('punkt_tab')
```

## Module Descriptions

### `generator.py`
Generates 500 synthetic JSON payloads with controlled random variability (`seed=42` for full reproducibility). Each of the five categories produces payloads with distinct structures to represent real-world AI use cases:

| Category | Payload structure | Key fields |
|---|---|---|
| `chat` | Chat session with role/content messages | `session_id`, `model`, `temperature`, `messages[]` |
| `rag` | Query with retrieved document chunks | `query`, `top_k`, `rerank`, `chunks[]` |
| `agent` | Agent config with available tools | `agent_id`, `goal`, `tools_available[]` |
| `finetune` | Conversation pair with quality metadata | `example_id`, `split`, `conversation[]`, `labels[]` |
| `prompt` | Prompting technique with variables and constraints | `technique`, `variables{}`, `constraints{}` |

### `converter.py`
Implements TOON v1.2 with compact serialization. Applies the following conversion rules:

| JSON type | TOON representation |
|---|---|
| `{}` object | `<>` |
| `true` | `+` |
| `false` | `-` |
| `null` | `~` |
| `"key"` (string key) | `key` (no quotes) |
| `[1, 2, 3]` scalar list | `[1 2 3]` (space-separated) |

Simple objects with 6 or fewer scalar fields are serialized on a **single line**. This is critical for efficiency: unrestricted multiline formatting increases token consumption instead of reducing it — an issue discovered and corrected during the experiment.

### `token_counter.py`
Uses `tiktoken` with the `cl100k_base` tokenizer to count tokens without making any network or model call. Runs 100% locally on serialized strings. This is how token counts are measured in the paper without requiring an API.

### `inference.py`
Communicates with Ollama through its local REST API at `http://localhost:11434`. Sends each payload in both formats (JSON and TOON) with `temperature=0.0` for deterministic output, and measures the latency of each call. **No API key required.**

### `evaluator.py`
Computes BLEU-4 (via `nltk`) and ROUGE-L (via `rouge-score`) by comparing the LLM response generated from the JSON payload (reference) against the response generated from the equivalent TOON payload (hypothesis). High ROUGE-L indicates semantic equivalence even when exact wording differs.

### `run_experiment.py`
Main orchestrator. Coordinates all five modules above, displays a real-time progress bar per category, and writes `results/results.csv` with one row per processed instance.

### `report.py`
Reads `results/results.csv` and produces a formatted console summary plus three PNG charts saved to `results/charts/`. Requires `matplotlib`.

## Running the Experiment

> **Before running `sample` or `full` mode:** make sure Ollama is running in the background.  
> On Windows, open a terminal and verify with `ollama list`. If nothing appears, start Ollama from the system tray or run `ollama serve`.

### Step 1 — Verify the converter and token counter (no LLM needed)

```bash
python converter.py
python token_counter.py
```

Both scripts include a built-in demo. If they print output without errors, your setup is correct.

### Step 2 — Tokens-only mode (2–3 minutes, no LLM)

Generates all 500 JSON payloads, converts them to TOON, and counts tokens. The fastest way to verify the full pipeline without inference.

```bash
python run_experiment.py --mode tokens
```

### Step 3 — Sample mode with LLM inference (15–20 minutes)

Runs inference on 10 instances per category (50 total). Recommended before committing to the full run.

```bash
python run_experiment.py --mode sample --n 10
```

You can increase the sample size:

```bash
python run_experiment.py --mode sample --n 25
python run_experiment.py --mode sample --n 50
```

### Step 4 — Full experiment: 500 instances (~90 minutes)

Runs inference on all 500 instances across the five categories.

```bash
python run_experiment.py --mode full
```

### Step 5 — Generate the report and charts

```bash
python report.py
```

Prints the results table to the console and saves three charts to `results/charts/`.

## Results

After running the experiment, `results/results.csv` contains one row per instance with the following columns:

| Column | Description |
|---|---|
| `category` | Payload category: `chat`, `rag`, `agent`, `finetune`, or `prompt` |
| `file` | Name of the processed JSON file |
| `json_tokens` | Token count of the minified JSON payload |
| `toon_tokens` | Token count of the TOON payload |
| `reduction_pct` | Token reduction percentage — **positive means TOON wins** |
| `json_chars` | Character count of the JSON payload |
| `toon_chars` | Character count of the TOON payload |
| `latency_json_s` | Inference latency with JSON payload (seconds) |
| `latency_toon_s` | Inference latency with TOON payload (seconds) |
| `bleu4` | BLEU-4 score comparing JSON vs. TOON LLM responses |
| `rouge_l` | ROUGE-L score comparing JSON vs. TOON LLM responses |

> `latency_json_s`, `latency_toon_s`, `bleu4`, and `rouge_l` are empty in `--mode tokens` since no LLM inference is performed.

### Paper results (n=500, llama3.1:8b, cl100k_base)

| Category | Token reduction | Char reduction | BLEU-4 | ROUGE-L |
|---|---|---|---|---|
| Simple chat | −4.70% | 5.05% | 0.3740 | 0.5637 |
| RAG pipeline | +2.38% | 7.26% | 0.2184 | 0.4464 |
| Multi-tool agent | +10.09% | 14.17% | 0.3152 | 0.4969 |
| Fine-tuning | −1.80% | 6.67% | 0.3879 | 0.5733 |
| Prompt engineering | +12.92% | 15.88% | 0.3466 | 0.5205 |
| **Global** | **+3.78%** | **8.94%** | **0.328** | **0.520** |

**Key finding — H3:** TOON's token reduction advantage is proportional to the density of primitive types (booleans, nulls, and simple scalar objects) in the payload. Categories with high primitive density (agent, prompt) show consistent gains across 100% of instances; categories dominated by long strings (chat) show consistent losses.

## The TOON Format

TOON (Token-Optimized Object Notation) is a serialization format designed to reduce token consumption when passing structured data to LLMs. It is structurally equivalent to JSON but syntactically more compact.

**JSON:**
```json
{
  "agent_id": "planner_01",
  "active": true,
  "memory": null,
  "verbose": false,
  "tools": [
    {"name": "read_file", "enabled": true, "timeout_ms": 3000}
  ]
}
```

**TOON:**
```
agent_id:"planner_01"
active:+
memory:~
verbose:-
tools:[
  <name:"read_file" enabled:+ timeout_ms:3000>
]
```

The compact representation eliminates quotes from keys, replaces `true`/`false`/`null` with single characters, and collapses simple objects to a single line — all changes that reduce the number of tokens the tokenizer produces from the same semantic content.
